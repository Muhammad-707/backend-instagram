## Проверено живыми запросами

### Три бага softclub закрыты
- **#3 (userId из JWT):** лента eraj (41) ≠ лента daler (36) — из токена, не из query.
- **#4 (пагинация):** стр.1 [99,95,91,90,87] ≠ стр.2 [83,82,81,79,75].
- **#5 (< 300 мс):** feed?limit=20 → **среднее 41 мс, макс 46 мс**. SQL по EXPLAIN ANALYZE — **0.9 мс**.
- **#6 (автор комментария не null):** author — объект с userName/avatar.

### Остальное
- `POST /posts` (фото+видео) → 201: постер видео, фильтры per-media, музыка, локация, отметка; usesCount музыки +1
- Хэштеги: #travel+#Travel → один тег, кириллический #закат распознан; @упоминание + отметка → уведомления MENTION/TAG_POST
- Комментарии: ответ (parentId), лайк toggle, список с repliesCount; **чужой → 403**, свой → 200
- Лайк поста (toggle), /:id/likes, просмотр (**не удваивается**), избранное+коллекция → /profile/favorites
- Share: ссылка / в чат (чат + POST_SHARE) / в историю; report → 201

- Swagger: posts 22, всего 82 · build · lint · prisma validate → зелёные

## Решения (почему так, а не иначе)
- **Один POST_SELECT с include на все ленты** — автор/медиа/локация/музыка/отметки/хэштеги/счётчики за один запрос.
  Это лекарство от 21 секунды: старый API делал запрос на каждый пост (N+1).
- **Сортировка по id DESC**, не createdAt: id монотонен, курсор однозначен, btree по целым дешевле.
- **Замер только через 127.0.0.1**: localhost в Windows добавляет ~200 мс на резолв IPv6 (артефакт curl, не API).
- **Удалять комментарий может автор ИЛИ владелец поста** (как в IG). Остальные → 403.
- **Ответы — один уровень**: ответ на ответ прикрепляется к корневому комментарию.
- **View/like идемпотентны** (@@unique postId+userId).
- **Explore не показывает закрытые аккаунты** (кроме подписок) и заблокированных.
- **Откат при сбое заливки**: пост не создался → залитые файлы удаляются (как в Фазе 2).
- **toBool/toStringArray в DTO**: multipart шлёт строками.

## Фаза 0

### #1 — API падал при старте, если БД недоступна
- **Дата:** 2026-07-14
- **Где:** `src/prisma/prisma.service.ts` → `onModuleInit()`
- **Симптом:** `PrismaClientInitializationError: P1000 Can't reach database server` → процесс Node падал целиком, `/api/health` не отвечал вообще (curl → пусто, exit 7).
- **Причина:** `await this.$connect()` без обработки ошибки. Nest не мог инициализировать модуль → bootstrap падал.
- **Починка:** `$connect()` обёрнут в `try/catch` с логированием. API поднимается, а `/api/health` честно отвечает `{"status":"degraded","database":"down"}`.
- **Почему так, а не «пусть падает»:** health-check обязан **сообщать** о проблеме, а не молчать. Плюс без этого нельзя было проверить Swagger до готовности Docker.
- **Статус:** ✅ закрыт, проверен curl'ом.

## Фаза 2

### #5 — `P1000 Authentication failed`: порт 5432 занят локальным PostgreSQL
- **Дата:** 2026-07-15
- **Где:** окружение (не код) · `docker-compose.yml`, `.env`
- **Симптом:** контейнер `ig_postgres` — `healthy`, `psql` **внутри** контейнера работает, но `npx prisma migrate dev` с хоста → `P1000: Authentication failed ... credentials for (not available) are not valid`. Креды при этом верные.
- **Причина:** `netstat -ano | grep :5432` показал **два** слушателя: `com.docker.backend` (PID 10764) и **локально установленный Windows-сервис PostgreSQL** (PID 5148). Локальный Postgres перехватывал подключения раньше docker-proxy и отвергал пару `ig/ig_password` — она существует только в контейнере.
- **Починка:** контейнер публикуется на **5433** (`'${POSTGRES_HOST_PORT:-5433}:5432'`), `DATABASE_URL` на хосте → `localhost:5433`. Локальную установку не трогали. Внутри docker-сети сервис `api` ходит на `postgres:5432` — там ничего не менялось.
- **Урок:** «контейнер healthy» ≠ «хост достучится до контейнера». Проверять, кто ещё слушает порт.
- **Статус:** ✅ закрыт, миграция и seed прошли.

### #6 — ffmpeg/ffprobe отсутствуют в системе
- **Дата:** 2026-07-15
- **Где:** `src/storage/media.service.ts`
- **Симптом:** `ffmpeg: command not found` — `fluent-ffmpeg` без бинаря бесполезен (постер видео и длительность не снять).
- **Починка:** `npm i ffmpeg-static ffprobe-static` (+ `@types/ffprobe-static`), пути прописаны через `ffmpeg.setFfmpegPath()` / `setFfprobePath()`. Бинари приезжают npm-пакетом — системная установка и `PATH` не нужны, `docker compose` тоже не ломается.
- **Статус:** ✅ закрыт, ffmpeg 6.1.1 работает, постер снимается.

### #7 — Частичная загрузка оставляла бы мусор в S3
- **Дата:** 2026-07-15
- **Где:** `src/modules/upload/upload.service.ts`
- **Симптом:** потенциальный (поймали на этапе проектирования, не в проде): при заливке пачки из 10 файлов ошибка на 5-м оставила бы первые 4 навсегда лежать в bucket — на них никто уже не сошлётся.

- **Проверено:** пачка «хорошее фото + подделка» → `400`, число объектов в bucket не изменилось.
- **Статус:** ✅ закрыт.

## Фаза 3

### #8 — `/upload` был открыт анониму (закрыт)
- **Дата:** 2026-07-15
- **Где:** `src/app.module.ts`, `src/modules/upload/upload.controller.ts`
- **Симптом:** после Фазы 2 `POST /upload` и `DELETE /upload/:key` работали **без токена** — любой аноним мог залить 10 файлов по 100 МБ.
- **Починка:** `JwtAuthGuard` повешен **глобально** (`APP_GUARD`), всё закрыто по умолчанию, открывается точечно `@Public()`. В upload-контроллере не понадобилось ни одной правки логики — только снят `TODO` и добавлен `@ApiBearerAuth()`.
- **Почему глобально, а не `@UseGuards` на каждом контроллере:** при 137 endpoint'ах «забыть повесить guard» — вопрос времени. Глобальный запрет + явное открытие делает дыру невозможной по умолчанию.
- **Проверено:** `POST /upload` без токена → `401`; с токеном → `201`.
- **Статус:** ✅ закрыт.

### #9 — MailHog рвёт соединение, если nodemailer шлёт AUTH
- **Дата:** 2026-07-15
- **Где:** `src/mail/mail.service.ts`
- **Симптом:** при `auth: { user: '', pass: '' }` nodemailer всё равно отправляет команду AUTH, а MailHog её не поддерживает → соединение рвётся, письмо не уходит.
- **Починка:** `auth: user ? { user, pass } : undefined` — при пустом `SMTP_USER` (dev/MailHog) блок auth не отправляется вовсе. Тот же код с заполненным `SMTP_USER` работает с Gmail.
- **Статус:** ✅ закрыт, `SMTP connected`, письмо реально доставлено.

### #10 — resetToken был бы переиспользуем (поймано при проектировании)
- **Дата:** 2026-07-15
- **Где:** `src/modules/auth/auth.service.ts`
- **Симптом:** потенциальный. JWT по своей природе переиспользуем: получив `resetToken` один раз, им можно было бы менять пароль сколько угодно раз в течение 15 минут.
- **Починка:** `jti` токена кладётся в Redis (TTL 15 мин) и **удаляется при первом** `reset-password` (`del` вернёт 0 → `400`). Плюс claim `typ: 'reset'` — обычным access-токеном пароль не сменить.
- **Проверено:** повторный `reset-password` с тем же токеном → `400` «resetToken уже использован».
- **Статус:** ✅ закрыт.

## Фаза 4а

### #11 — dev-сервер упал: `npm run build` затёр `dist/` под работающим watch'ем
- **Дата:** 2026-07-15
- **Где:** окружение (не код)
- **Симптом:** после `npm run build` API умер: `Cannot find module './auth.controller'`, `requireStack: dist/modules/auth/auth.module.js`. `curl` → exit 7, HTTP 000.
- **Причина:** `npm run start:dev` (watch) и `nest build` пишут в **один и тот же `dist/`**. Build снёс папку, watcher подхватил полупустое состояние и упал.
- **Починка:** перезапуск `start:dev`. На будущее: не гонять `nest build` параллельно с работающим watch — либо остановить dev-сервер, либо собирать в отдельную папку.
- **Статус:** ✅ не баг кода, но время съело. Записано, чтобы не искать причину заново.

### #12 — «API портит UTF-8» — ЛОЖНАЯ тревога
- **Дата:** 2026-07-15
- **Симптом:** `POST /users/search-history` с текстом «закат в горах» → в ответе `U+FFFD` (заменяющие символы), сравнение `text === 'закат в горах'` → `false`.
- **Проверка:** тот же запрос через Node `fetch` (кодировка гарантированно UTF-8) → **round-trip корректен, `true`**.
- **Причина:** кириллицу ломал **сам `curl` в Windows-шелле** (кодировка аргумента `-d`), а не API. В PostgreSQL текст лежал правильно (13 символов = 13 букв).
- **Вывод:** проверять кириллицу только через Node/Postman, не через `curl` в этом шелле. `length` строки — быстрый способ отличить реальную порчу от кракозябр консоли.
- **Статус:** ✅ закрыт, бага нет.

### #13 — Пустая вкладка ≠ рабочая вкладка
- **Дата:** 2026-07-15
- **Где:** `src/modules/profile/profile.service.ts`
- **Симптом:** `favorites`, `reposts`, `saved-music`, `tagged` вернули `200` и **0 записей** — в seed таких данных просто нет. Ответ `200 []` не доказывает, что запрос верный: он одинаков и для правильного, и для сломанного `where`.
- **Проверка:** вставил записи в `Favorite`, `SavedMusic`, `Share`, `PostTag` прямо в БД → вкладки отдали **2 · 3 · 2 · 2**.
- **Вывод:** «тахмин манъ» касается и пустых ответов — если данных нет, их надо создать, а не считать endpoint рабочим.
- **Статус:** ✅ закрыт, все четыре вкладки проверены на реальных данных.

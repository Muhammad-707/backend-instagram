### #5 — `P1000 Authentication failed`: порт 5432 занят локальным PostgreSQL
- **Дата:** 2026-07-15
- **Где:** окружение (не код) · `docker-compose.yml`, `.env`
- **Симптом:** контейнер `ig_postgres` — `healthy`, `psql` **внутри** контейнера работает, но `npx prisma migrate dev` с хоста → `P1000: Authentication failed ... credentials for (not available) are not valid`. Креды при этом верные.
- **Причина:** `netstat -ano | grep :5432` показал **два** слушателя: `com.docker.backend` (PID 10764) и **локально установленный Windows-сервис PostgreSQL** (PID 5148). Локальный Postgres перехватывал подключения раньше docker-proxy и отвергал пару `ig/ig_password` — она существует только в контейнере.
- **Починка:** контейнер публикуется на **5433** (`'${POSTGRES_HOST_PORT:-5433}:5432'`), `DATABASE_URL` на хосте → `localhost:5433`. Локальную установку не трогали. Внутри docker-сети сервис `api` ходит на `postgres:5432` — там ничего не менялось.
- **Урок:** «контейнер healthy» ≠ «хост достучится до контейнера». Проверять, кто ещё слушает порт.
- **Статус:** ✅ закрыт, миграция и seed прошли.

### #8 — `/upload` был открыт анониму (закрыт)
- **Дата:** 2026-07-15
- **Где:** `src/app.module.ts`, `src/modules/upload/upload.controller.ts`
- **Симптом:** после Фазы 2 `POST /upload` и `DELETE /upload/:key` работали **без токена** — любой аноним мог залить 10 файлов по 100 МБ.
- **Починка:** `JwtAuthGuard` повешен **глобально** (`APP_GUARD`), всё закрыто по умолчанию, открывается точечно `@Public()`. В upload-контроллере не понадобилось ни одной правки логики — только снят `TODO` и добавлен `@ApiBearerAuth()`.
- **Почему глобально, а не `@UseGuards` на каждом контроллере:** при 137 endpoint'ах «забыть повесить guard» — вопрос времени. Глобальный запрет + явное открытие делает дыру невозможной по умолчанию.
- **Проверено:** `POST /upload` без токена → `401`; с токеном → `201`.
- **Статус:** ✅ закрыт.

### #9 — MailHog рвёт соединение, если nodemailer шлёт AUTH
- **Дата:** 2026-07-15
- **Где:** `src/mail/mail.service.ts`
- **Симптом:** при `auth: { user: '', pass: '' }` nodemailer всё равно отправляет команду AUTH, а MailHog её не поддерживает → соединение рвётся, письмо не уходит.
- **Починка:** `auth: user ? { user, pass } : undefined` — при пустом `SMTP_USER` (dev/MailHog) блок auth не отправляется вовсе. Тот же код с заполненным `SMTP_USER` работает с Gmail.
- **Статус:** ✅ закрыт, `SMTP connected`, письмо реально доставлено.

### #10 — resetToken был бы переиспользуем (поймано при проектировании)
- **Дата:** 2026-07-15
- **Где:** `src/modules/auth/auth.service.ts`
- **Симптом:** потенциальный. JWT по своей природе переиспользуем: получив `resetToken` один раз, им можно было бы менять пароль сколько угодно раз в течение 15 минут.

- **Проверено:** повторный `reset-password` с тем же токеном → `400` «resetToken уже использован».
- **Статус:** ✅ закрыт.

### #12 — «API портит UTF-8» — ЛОЖНАЯ тревога

### #16 — `GET /posts/my?archived=true` → 400 (query-параметр отвергался валидацией)
- **Дата:** 2026-07-15
- **Где:** `src/modules/posts/posts.controller.ts`, `dto/post.dto.ts`
- **Симптом:** `/posts/my?archived=true` → `400 Validation failed`.
- **Причина:** метод принимал `@Query() dto: CursorDto`, а `archived` в этом DTO не описан.
  Глобальный `ValidationPipe({ forbidNonWhitelisted: true })` отвергает любой незнакомый query-параметр.
- **Починка:** создан `MyPostsQueryDto extends CursorDto` с полем `archived` (`@Transform(toBool)`).
- **Урок:** при `forbidNonWhitelisted: true` **каждый** query-параметр должен быть полем DTO, иначе `400`.
- **Статус:** ✅ закрыт, `?archived=true` → `200`.

### #17 — Ложные 250 мс на ленте: резолв `localhost` в Windows, не API
- **Дата:** 2026-07-15
- **Где:** окружение (не код)
- **Симптом:** `GET /posts/feed` через `http://localhost:3000` замерялся в 250–300 мс — на грани требования < 300 мс.
- **Диагностика:** даже пустой `/api/health` через `localhost` отвечал 210 мс. `EXPLAIN ANALYZE` показал
  выполнение SQL ленты за **0.9 мс**. Значит время уходило не на запрос.
- **Причина:** curl на Windows сначала пробует IPv6 (`::1`) для `localhost`, ловит таймаут, откатывается на IPv4 — отсюда ~200 мс.
- **Проверка:** через `http://127.0.0.1:3000` та же лента → **41 мс среднее, 46 мс максимум**.
- **Урок:** мерить производительность только через `127.0.0.1`; `localhost` в этом окружении добавляет фиктивные 200 мс.
- **Статус:** ✅ не баг кода. Реальная скорость ленты — 41 мс, цель < 300 мс достигнута с запасом.

## Фаза 9

### #20 — Сервер падал при disconnect сокета: `this.server.of is not a function`
- **Дата:** 2026-07-15
- **Где:** `src/modules/realtime/realtime.gateway.ts` → `handleDisconnect`
- **Симптом:** при закрытии сокет-клиента весь процесс падал (`TypeError`), API умирал (`ECONNREFUSED`).
- **Причина:** в namespace-шлюзе (`namespace: '/rt'`) `@WebSocketServer()` инжектит **Namespace**, а не корневой `Server`.
  У Namespace нет метода `.of()`, поэтому `this.server.of('/rt')` рушился.
- **Починка:** adapter берём через namespace самого сокета — `client.nsp.adapter.rooms.get(...)` (корректно типизировано,
  в отличие от `this.server.adapter`, где TS видит перегруженную функцию).
- **Урок:** в namespace-гейтвее `@WebSocketServer()` — это Namespace. Для проверки комнат использовать `client.nsp.adapter`.
- **Статус:** ✅ закрыт, disconnect не роняет процесс.

### #21 — Запрос на переписку не переоткрывался после DECLINED
- **Дата:** 2026-07-15
- **Где:** `src/modules/chat/chat.service.ts` → `createOrGet`
- **Симптом:** после `decline` повторный `POST /chats` от того же неподписанного юзера НЕ возвращал заявку в «Запросы»
  (тест ждал 1 строку в `/chats/requests`, получил 0).
- **Причина:** при уже существующем чате был ранний `return { existed: true, isRequest: false }`, который пропускал
  обновление `MessageRequest` в PENDING. А чат между этими двумя уже существовал с первой (отклонённой) заявки.
- **Починка:** логику переписали — сначала находим/создаём чат, затем **независимо** решаем про запрос: если не подписан
  и запрос не ACCEPTED, `upsert` в PENDING. `@@unique(from,to)` гарантирует одну строку (антиспам, ТЗ §5.10).
- **Проверка:** decline → повторная заявка → в `/chats/requests` ровно **1** строка.
- **Статус:** ✅ закрыт.
